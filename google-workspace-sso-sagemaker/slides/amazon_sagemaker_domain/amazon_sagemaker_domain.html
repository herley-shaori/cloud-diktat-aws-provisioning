Amazon SageMaker AI Domain

Managed ML environment with SSO integration and private VPC networking

Amazon SageMaker

Fully managed machine learning service

Amazon SageMaker AI Domain is a logical container that groups users, notebooks, and ML resources together. When combined with IAM Identity Center SSO, users from Google Workspace can securely access SageMaker Studio and Jupyter notebooks without managing separate AWS credentials.

What is SageMaker Domain?

A managed environment for machine learning workflows

💻

SageMaker Studio

A web-based IDE for ML development. Users can write code, train models, and deploy endpoints - all from a browser. Each user gets their own JupyterServer app.

👤

User Profiles

Each user has a User Profile that defines their execution role, security groups, and personal settings. Profiles are isolated from each other.

🔒

Authentication Modes

Supports two auth modes: IAM (for programmatic access) and SSO (for federated identity via IAM Identity Center).

📁

Amazon EFS Storage

Each domain automatically creates an Amazon EFS file system. User home directories and notebooks are persisted here, surviving instance restarts.

🌐

Network Isolation

Domains can run in PublicInternetOnly or VpcOnly mode. VpcOnly provides maximum security by keeping all traffic within your VPC.

⚙

Execution Role

An IAM Role that SageMaker assumes when running notebooks and training jobs. Controls what AWS resources the ML workloads can access.

💡 Key Point: A SageMaker Domain is like a "workspace" that contains everything your data science team needs - user profiles, notebooks, compute resources, and shared storage.

Private Mode (VpcOnly)

Maximum security with all traffic routed through your VPC

🔒 What is VpcOnly Mode?

In VpcOnly mode, SageMaker Domain runs entirely within your VPC. There is no direct internet access - all AWS service communication happens through VPC Endpoints. This is ideal for regulated industries and sensitive data.

✔ Benefits
  • No data leaves your VPC
  • Full network control
  • Meets compliance requirements
  • Can integrate with on-premises
⚠ Requirements
  • VPC Endpoints for AWS services
  • Private subnets
  • Proper security groups
  • NAT Gateway (optional, for updates)
VPC: 10.0.0.0/16

sagemaker-vpc

Core Infrastructure Components

The building blocks that make VpcOnly mode work

🔐

Security Groups

Two security groups control network access:

sagemaker-domain-sg

Attached to SageMaker. Allows all outbound traffic and permits internal communication (self-referencing).

vpc-endpoints-sg

Attached to VPC Endpoints. Allows HTTPS (443) from SageMaker security group and private subnets.

🔌

VPC Endpoints

Private connections to AWS services (no internet required):

S3 (Gateway) SageMaker API SageMaker Runtime SageMaker Studio STS CloudWatch Logs ECR API ECR DKR Service Catalog

💡 Without these endpoints, SageMaker cannot access AWS services in VpcOnly mode.

📁

Amazon EFS

Elastic File System is automatically created by SageMaker Domain:

  • Stores user home directories
  • Persists Jupyter notebooks
  • Shared across all user profiles
  • Survives instance stop/start
  • Mounted at /home/sagemaker-user
⚙

IAM Execution Role

The role SageMaker assumes when running workloads:

sagemaker-execution-role

Attached policies:

AmazonSageMakerFullAccess AmazonS3ReadOnlyAccess

This role determines what AWS resources notebooks and training jobs can access.

🌐

VPC & Subnets

The foundational network infrastructure for SageMaker Domain:

sagemaker-vpc (10.0.0.0/16)

DNS hostnames & DNS support enabled for VPC endpoint resolution.

Public Subnets (2x) Private Subnets (2x) Multi-AZ

SageMaker runs in private subnets with no direct internet access.

🚨

NAT Gateway

Enables outbound-only internet access from private subnets:

  • Download packages from PyPI, conda
  • Pull container images from Docker Hub
  • Access external APIs and datasets
  • Software updates for notebooks
Route: 0.0.0.0/0 → NAT Gateway

Private route table directs all external traffic through NAT in public subnet.

Google Workspace SSO to SageMaker

How Terraform configures the complete authentication flow

1
🏗
AWS Organizations & OU
organizational-unit.tf creates a "Data Scientist" Organizational Unit under the root. This provides logical grouping and can be used for SCPs (Service Control Policies).
2
👥
Identity Store Group & Permission Set
sso.tf creates a DataScientists group in IAM Identity Center and a DataScientist permission set with SageMakerFullAccess + S3ReadOnlyAccess policies.
3
👤
User Lookup from SCIM
The aws_identitystore_user data source looks up the user herley@cloud-diktat.info who was automatically provisioned from Google Workspace via SCIM synchronization.
4
🔗
Group Membership & Account Assignment
User is added to the DataScientists group. The group is then assigned to the AWS account with the DataScientist permission set - enabling AWS Console access via SSO portal.
5
🤖
SageMaker Domain with SSO Auth
sagemaker.tf creates the domain with auth_mode = "SSO". The domain runs in VpcOnly mode with private subnets and depends on all VPC endpoints being ready.
6
📝
User Profile Linked to SSO Identity
A aws_sagemaker_user_profile is created with single_sign_on_user_value = "herley@cloud-diktat.info". Only this SSO user can claim and use this SageMaker profile.
7
🎯
Application Assignment
aws_ssoadmin_application_assignment grants the DataScientists group access to the SageMaker Studio SSO application. Users see SageMaker in their AWS access portal.

✔ Result: A user from Google Workspace logs in via the AWS access portal, clicks on SageMaker Studio, and is automatically directed to their personal Jupyter environment - no AWS credentials needed!
